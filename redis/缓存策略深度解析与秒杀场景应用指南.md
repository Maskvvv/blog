# 缓存策略深度解析与秒杀场景应用指南

## 1. 三种核心缓存策略对比

在分布式系统中，缓存与数据库的一致性维护是核心难题。常见的三种策略为 **Cache Aside**、**Read/Write Through** 和 **Write Behind**。

### 1.1 Cache Aside Pattern（旁路缓存模式）
**最常用、最推荐的通用策略**，尤其适用于 Redis + MySQL 架构。

- **核心逻辑**：应用程序直接同时操作缓存和数据库，缓存作为“旁路”存在。
- **读流程**：
  1. 先读缓存。
  2. 命中则返回；未命中则读数据库，写入缓存后返回。
- **写流程**：
  1. **先更新数据库**。
  2. **再删除缓存**（推荐删除而非更新，避免并发脏数据）。
- **优点**：
  - 实现简单，逻辑由应用完全控制。
  - 缓存仅存储热点数据，节省内存。
- **缺点**：
  - 应用层代码耦合缓存逻辑。
  - 极端并发下可能存在短暂不一致（通常通过设置过期时间缓解）。
- **适用场景**：绝大多数业务场景，特别是读多写少、对一致性要求非强实时的场景。

### 1.2 Read/Write Through Pattern（读写穿透模式）
依赖**支持该特性的缓存中间件**（如 Caffeine, Guava Cache），应用层对数据库透明。

- **核心逻辑**：应用层只与缓存交互，缓存组件内部封装数据库操作。
- **读流程 (Read Through)**：
  1. 应用向缓存请求数据。
  2. 未命中时，**缓存组件内部**自动调用 `Loader` 加载数据库数据并回填。
- **写流程 (Write Through)**：
  1. 应用向缓存写入数据。
  2. **缓存组件同步地**将数据写入数据库，成功后才返回。
- **优点**：
  - 应用层代码简洁，无需关心同步逻辑。
  - 数据一致性较好（同步写库）。
- **缺点**：
  - 写性能较差（受限于数据库 IO 延迟）。
  - 需要特定框架支持，通用性不如 Cache Aside。
- **适用场景**：读多写少，希望简化代码或使用本地缓存框架的场景。

### 1.3 Write Behind Pattern（异步缓存写入/回写模式）
Write Through 的优化版，牺牲一致性换取极致写性能。

- **核心逻辑**：应用只写缓存，缓存组件**异步批量**刷新数据到数据库。
- **写流程**：
  1. 应用写入缓存，立即返回成功。
  2. 缓存组件在后台按策略（定时、定数量等）异步写入数据库。
- **优点**：
  - **写性能极高**：应用不等待数据库 IO。
  - 可合并多次写操作，减少数据库压力。
- **缺点**：
  - **数据丢失风险**：缓存宕机可能导致未刷库的数据永久丢失。
  - 一致性最差：数据库数据存在延迟。
- **适用场景**：写极频繁、允许少量丢失或不强一致的场景（如计数器、点赞数、日志缓冲）。

### 1.4 策略对比总结表

| 特性         | Cache Aside (旁路)           | Read/Write Through (穿透)        | Write Behind (回写)            |
| :----------- | :--------------------------- | :------------------------------- | :----------------------------- |
| **维护者**   | **应用程序** (手动编码)      | **缓存中间件** (自动处理)        | **缓存中间件** (自动处理)      |
| **读逻辑**   | 应用查缓存 -> miss则查库回填 | 应用查缓存 -> miss则缓存组件查库 | 同 Read Through                |
| **写逻辑**   | 先更库 -> 再删缓存           | 写缓存 -> 缓存组件**同步**写库   | 写缓存 -> 缓存组件**异步**写库 |
| **一致性**   | 较高                         | 高 (强一致)                      | 低 (最终一致，有丢失风险)      |
| **写性能**   | 中等                         | 低 (同步阻塞)                    | **极高** (无DB阻塞)            |
| **实现难度** | 低 (通用)                    | 中 (需框架支持)                  | 高 (需处理异常/丢失)           |
| **典型应用** | **Redis + MySQL 主流架构**   | Caffeine, Guava Cache            | 高频统计, 日志缓冲             |

## 2. 秒杀业务场景分析

### 2.1 核心结论
**秒杀业务绝对不能直接使用 `Write Behind` 策略作为库存扣减的核心逻辑。**

若在秒杀中直接对数据库库存使用 Write Behind，会导致**超卖（Overselling）**，这是电商业务的致命事故。

### 2.2 为什么不能用 Write Behind？

#### A. 超卖风险（致命）
- **机制缺陷**：Write Behind 是“先改缓存，异步刷库”。
- **故障场景**：
  1. 库存 100 件，高并发下 Redis 成功扣减了 100 次，返回 100 个“抢购成功”。
  2. 这 100 条扣减记录还在 Redis 队列中等待异步写入 MySQL。
  3. **突发情况**：此时 Redis 节点宕机，或异步线程崩溃。
  4. **后果**：MySQL 中可能只收到了 50 条甚至 0 条扣减记录。
  5. **结局**：用户端显示成功 100 单，实际库存只扣了 50，导致 50 单无法发货，引发严重客诉。

#### B. 实时校验失效
- 秒杀依赖精确的库存判断。Write Behind 导致数据库库存数据滞后，无法作为可靠的校验依据；若仅依赖缓存，一旦缓存故障，数据一致性无法保障。

### 2.3 秒杀业务的正确缓存策略

秒杀架构通常采用 **“Redis 原子扣减 + 消息队列削峰”** 的组合方案。

#### 标准流程：
1.  **库存预热**：
    - 活动前将库存加载至 Redis。
2.  **原子扣减（关键步骤）**：
    - 使用 **Lua 脚本** 执行 `decr` 操作。
    - **原理**：Redis 执行 Lua 脚本是单线程原子的，确保高并发下库存不会减成负数（杜绝超卖）。
    - *注意：这是在内存中同步完成的，不是异步回写。*
3.  **异步下单（削峰）**：
    - Lua 脚本扣减成功后，发送消息到 **消息队列（MQ）**。
    - 立即返回用户“排队中”或“抢购成功”。
    - *区别点：这里是异步创建订单，而非异步扣减库存。库存已在 Redis 中确认为扣减成功。*
4.  **数据库落库**：
    - 消费者从 MQ 拉取消息，以可控速度写入 MySQL，并进行二次校验（乐观锁）。

### 2.4 Write Behind 在秒杀中的有限用途
虽然不能用于库存扣减，但可用于非核心场景：
- **非关键统计**：浏览量、点赞数、热度排名（允许少量丢失）。
- **行为日志**：用户点击流记录。

## 3. 选型建议总结

| 场景                  | 推荐策略               | 理由                                                         |
| :-------------------- | :--------------------- | :----------------------------------------------------------- |
| **通用业务**          | **Cache Aside**        | 灵活、通用、平衡性能与一致性，Redis 首选。                   |
| **简化代码/本地缓存** | **Read/Write Through** | 利用框架特性减少样板代码，适合读多写少。                     |
| **高频统计/日志**     | **Write Behind**       | 追求极致写性能，容忍少量数据丢失。                           |
| **秒杀/库存扣减**     | **Redis Lua + MQ**     | **严禁使用 Write Behind**。必须保证内存原子扣减防超卖，配合 MQ 异步落库。 |

> **核心原则**：涉及资金、库存等核心资产的业务，必须保证操作的**原子性**和**可靠性**，严禁为了性能牺牲数据的一致性安全。
